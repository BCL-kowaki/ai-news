# 事業頭脳・戦略メモ

> 最終更新: 2026-07-15

## 位置づけ

これは**個人用ツール**であり、収益化を目的としたプロダクトではない。
「AIの最新情報を、コストゼロで、手間なく受け取る」という自分の課題を解くのが目的。
戦略の軸は「事業成長」ではなく **維持コストゼロの継続**。

## LINE → Slack に変更した判断（重要な意思決定）

当初LINE想定だったが、着手前にSlackと比較して**Slackに変更**した。理由:

| 観点 | LINE | Slack（採用） |
|---|---|---|
| 無料枠 | 月200通の上限。設計全体がこの制約に縛られる | 送信数は実質無制限。上限を気にしなくてよい |
| 配信頻度 | 無料枠を守るため1日3回にまとめる必要 | 毎時でも無料（今は通知過多を避け1日3回） |
| セットアップ | トークン・シークレット・Webhook・userID取得・署名検証 | **Webhook URL 1個の発行だけ** |
| リンク表示 | 素のテキスト | URLの自動プレビュー・検索が標準 |
| 実装の複雑さ | 無料枠ガード・Reply・署名検証が必要 | それらが不要で**むしろシンプル** |

唯一のトレードオフは「普段どちらを開くか」。ユーザーは普段Slackを開くためSlackが最適と判断。

## この構成の強み・弱み

**強み**
- 完全無料で自動運用。放置していても毎日届く
- X APIを使わない（月$200を回避）。RSSは無料で一次情報源をカバー
- 要約しないのでAI API課金ゼロ、情報の改変リスクもゼロ
- Slackの検索・履歴で過去記事を遡れる（専用の一覧機能を作らなくてよい）

**弱み / リスク**
- RSSソースの停止・仕様変更で収集が止まる → 7ソースに分散（1つ落ちても他が動く）。実際VentureBeatが更新停止していたため差し替え済み
- Google News RSSはノイズ・重複が多い（1日100件超）→ 1回の配信件数に上限を設けて対処。多すぎるようならソースを絞る
- GitHub Actionsのスケジュール遅延（数分〜十数分）→ ニュース用途では実害なし
- Neon Freeは無アクセスで自動停止するが、毎時fetchがあるため起きにくい
