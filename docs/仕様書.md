# 仕様書

> 実装と同期させる。機能を変えたらここも更新する。
> 最終更新: 2026-07-15

## 目的

AI関連ニュースをRSSで自動収集し、**Slack**の指定チャンネルに自動配信する個人用システム。
要約はしない（Claude API等の外部AI呼び出しはゼロ）。タイトル＋URLをそのまま配信する。

### なぜSlackなのか（LINEから変更）

当初はLINEを想定していたが、LINE Messaging APIの無料枠（月200通）に設計全体が縛られる欠点があった。
Slack Incoming Webhookは送信数の上限が実質無く、セットアップもWebhook URL 1個の発行だけで済み、
URLの自動プレビューや検索も標準で使えるため、この用途では上位互換と判断してSlackに変更した。
（詳細な比較は docs/事業頭脳・戦略メモ.md）

## 全体の流れ

```
[毎時] GitHub Actions → /api/cron/fetch → 各RSS取得 → 新着だけ Article(pending) に保存
[1日3回] GitHub Actions → /api/cron/broadcast → pending記事をまとめ Slackへ投稿 → sent に更新
```

過去記事はSlackのチャンネルにそのまま流れて残るため、「今すぐ見る」的な仕組みは不要
（Slackの検索・スクロールで遡れる）。

## 技術スタック

- Next.js 14 (App Router) / TypeScript
- Prisma + Neon PostgreSQL
- Tailwind CSS
- ホスティング: Vercel
- 定期実行: **GitHub Actions**（Vercel Hobbyのcron制限を回避。理由は下記）
- メッセージング: **Slack Incoming Webhook**

## なぜ定期実行がGitHub Actionsなのか

Vercel Hobby（無料）プランのCronは **1日1回まで** という制限があり、
毎時のRSS収集や1日3回の配信は**デプロイ時にエラーになる**（公式ドキュメントで確認）。
そのため定期実行はGitHub Actions（無料・分単位指定可）から各エンドポイントをHTTPで叩く方式にした。

Vercel Proに切り替える場合は `vercel.json` に以下を追記し、GitHub Actions側のスケジュールを止める:

```json
{
  "crons": [
    { "path": "/api/cron/fetch", "schedule": "0 * * * *" },
    { "path": "/api/cron/broadcast", "schedule": "0 23,4,10 * * *" }
  ]
}
```
（VercelのcronはUTC。23/4/10時UTC = JST 8/13/19時）

## DBスキーマ

`prisma/schema.prisma` 参照。3テーブル:

- **Source** … RSSソース。`src/lib/sources.ts` の定義から収集ジョブ実行時に自動同期される（定義から消したソースは自動でactive=falseになり収集対象から外れる）
- **Article** … 収集記事（title + url のみ）。`url` が一意キーで重複排除。status = pending / sent / skipped
- **SendLog** … 配信の記録。1レコード = 配信1回。稼働確認・統計用（無料枠カウントには使わない）

## API仕様

### GET/POST `/api/cron/fetch`（認証: Bearer CRON_SECRET）
毎時実行。sources同期 → 各RSS取得 → 48時間以内の新着URLだけ `Article(pending)` に保存。Slackは呼ばない。
戻り: `{ ok, inserted, pending, details[] }`

### GET/POST `/api/cron/broadcast`（認証: Bearer CRON_SECRET）
1日3回実行。処理順:
1. pending記事を新しい順に取得
2. 1回の上限（MAX_ARTICLES_PER_BROADCAST=30件）まで選抜
3. 10件ごとに分割し、Slackへ順に投稿（レート制限に配慮して間隔を空ける）
4. 送った記事を sent に更新
5. 上限超過で見送った記事のうち48時間より古いものを skipped にする（滞留防止）
6. SendLogに配信実績を記録

戻り: `{ ok, sent, messages, pendingLeft, expired }`

### GET `/`（ダッシュボード）
未送信/送信済み件数・今月の配信回数・最終配信・ソース数を表示。個人用のため noindex。

## 設定値（すべて `src/lib/config.ts` に集約）

| 定数 | 値 | 意味 |
|---|---|---|
| MAX_ARTICLES_PER_BROADCAST | 30 | 1回の配信で流す記事の上限 |
| ARTICLES_PER_MESSAGE | 10 | 1メッセージあたりの記事数（超過分は分割） |
| SLACK_MESSAGE_INTERVAL_MS | 1100 | 分割送信の間隔（レート制限対策） |
| FETCH_WINDOW_HOURS | 48 | 収集対象の鮮度 |

## セキュリティ

- Cronエンドポイント: `CRON_SECRET` によるBearer認証。**未設定なら常に拒否**（フェイルセーフ）
- 秘密情報（Webhook URL・DB接続・CRON_SECRET）はすべて環境変数。コードに書かない（`.env` は `.gitignore`済み）
- RSS由来のURLは http/https のみ許可（`javascript:` 等を拒否）
- Slackリンク記法に載せるタイトルは `< > &` をエスケープ（メッセージ崩れ・インジェクション防止）
- 認証比較は定数時間比較（timingSafeEqual）でタイミング攻撃を防ぐ
